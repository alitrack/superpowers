# 企业研究资产工作台设计

## 摘要

这个项目不应该被定义成“另一个通用 AI 工作空间”。它第一阶段最可信、最有价值的切入口，是：

**面向企业战略 / 战投团队的研究资产工作台。**

产品外层可以借鉴 Flowith 这类 AI 工作空间的吸引力，例如：

- 工作空间而不是聊天框
- 多线程探索而不是线性对话
- 知识、探索、执行放在同一个环境里
- 有 Agent 在后台持续推进任务

但真正的产品壁垒，不应该来自“更像 Flowith”，而应该来自：

**把研究过程沉淀成可复用的组织资产。**

一句话定义：

> 这是一款漂亮的 AI 原生研究工作空间，但它最核心的价值不是更快给答案，而是把战略问题如何被定义、拆解、验证、判断和收敛，沉淀成团队可以继承的研究资产。

## 为什么是这个方向

### 为什么不应该做成泛化 AI 工作空间

通用 AI 工作空间通常会在广度上竞争：

- 更多模型
- 更多 Agent
- 更强并发
- 更大的上下文
- 更丰富的通用创作能力

这些点有演示吸引力，但不一定能在高价值企业团队内部形成长期留存。

企业战略 / 战投团队最痛的问题，不是“没有 AI 可用”，而是：

- 同类研究不断重复做
- 关键假设没有被结构化保留
- 证据散落在聊天、文档、PPT、飞书和个人脑子里
- 为什么会得出某个结论无法被完整继承
- 换人之后，研究理解和判断链条会明显断层

所以第一阶段最应该解决的，不是“更快生成结论”，而是：

**让一次性研究劳动变成可复用的团队资产。**

### 为什么这条路更值得投

从投资人视角，这条路比“讲一个大而全的 AI workspace 故事”更扎实，因为它同时具备：

- 清晰的高价值客户
- 足够深的专业痛点
- 组织级复利
- 从单点产品长成平台的明确扩张路径

这个产品真正改变的，不是某一次输出的速度，而是企业战略研究的生产方式。一旦团队开始积累：

- 问题树
- 假设结构
- 证据链
- 判断路径
- 结论演化
- 可复用研究模板

迁移成本和留存都会自然增强。

## 与 Flowith 的关系

Flowith 的公开产品说明证明了一点：市场已经接受“AI 工作空间不等于聊天框”。因此，我们应该借它的工作空间心智，但不能复制它的产品边界。

可参考的公开材料：

- https://flowith.io/
- https://doc.flowith.io/
- https://doc.flowith.io/other-questions/subscriptions-and-billing
- https://flowith.io/blog/knowledge-garden/

### 应该借鉴的是工作空间形态，不是产品范围

应该借鉴的，是：

- `workspace` 而不是线性聊天的交互心智
- 大画布、多线程探索和持续推进的工作方式
- 知识沉淀与执行动作在同一环境中的衔接
- Agent 作为持续工作的协同单元，而不是一次性回答器

不应该照抄的，是：

- 泛创作平台定位
- 通用 Agent 市场
- 以模型数、并发数、套餐档位为核心的价值叙事
- 弱语义、强自由度的通用白板

### 对比表：Flowith 与本产品 V1

下表基于 Flowith 的公开资料理解其方向，不涉及其未公开实现。

| 维度 | Flowith（基于公开资料的理解） | 本产品 V1 |
| --- | --- | --- |
| 核心对象 | 通用工作空间、知识花园、任务与上下文游乐场 | 以研究问题为根的研究资产包，内部对象严格限定为问题、假设、证据、判断、结论、资产包 |
| 画布语义 | 强工作空间感，适合多线程探索 | 保留大画布吸引力，但节点具备研究语义和约束，不是自由白板 |
| Agent 角色 | 帮用户并行探索、持续处理任务 | 在研究过程中辅助拆解、找证据、提炼判断，但不能替代最终专业判断 |
| 结果形态 | 通用知识与任务沉淀 | 可发布、可版本化、可复用、可审计的研究资产包 |
| 证据与判断 | 公开资料未体现强约束分离 | 证据、判断、结论是不同对象，且必须按提升路径引用 |
| 复用方式 | 以工作空间与知识积累为主 | 以模板、克隆、分支复盘、跨项目导入已发布资产为主 |
| 权限与治理 | 面向通用使用者的账户与订阅体系 | 面向企业团队的角色权限、发布门禁、操作审计和跨团队共享边界 |

### V1 必须形成的具体差异机制

差异不在“更像 Flowith”，而在下面这些必须落地的机制：

1. **证据与判断强分离**
   每条 `Judgment` 必须引用一条或多条 `Evidence`，不能直接凭主观文本跳到结论。
2. **研究资产包发布机制**
   工作区里的临时探索与团队可复用资产分离，只有满足发布条件的内容才进入已发布资产包。
3. **分支保留与复盘机制**
   被否定或搁置的分支不被直接删除，而是以 `Parked` / `Superseded` 方式保留，供后续复盘。
4. **受约束的 Agent 协作机制**
   Agent 能建议、汇总、提示风险，但不能越过人工审核直接发布资产、核验证据或改权限。

## 产品定位

### 产品类别

建议把产品类别固定为：

**企业研究资产工作台**

而不是：

- 头脑风暴工具
- AI 报告生成器
- 通用 AI 画布
- 带 Agent 的知识库

### 第一阶段 ICP

第一阶段只服务一个足够聚焦的切口：

- **客户类型：** 中大型企业内部的战略 / 战投团队
- **团队规模：** 5 到 30 人
- **工作特征：** 每月持续做立项评估、行业扫描、标的研究、战略专题
- **组织要求：** 研究结果要进入管理层汇报、投委会或年度战略讨论，必须可复盘、可追溯、可交接

第一阶段不优先覆盖：

- 泛知识工作者
- 高实时协作的咨询交付团队
- 依赖重型跨系统执行编排的运营团队

### 核心价值主张

核心主张建议固定为：

**让战略研究过程可见、可复用、可累积。**

更展开一点，可以表达成：

- 让问题拆解不再丢失
- 让证据链不再散落
- 让判断过程可以复盘
- 让同类项目不再从零开始
- 让研究结论成为组织资产，而不只是个人输出

## 产品原则

第一版建议遵循以下原则。

### 1. 外层要漂亮，内层要严格

界面应该让用户一眼感到“这不是普通 chat”，但内部结构必须保持强语义和强边界，不能沦为纯展示型白板。

### 2. 先帮助定义问题，再帮助形成答案

战略研究里最贵的错误，常常来自问题定义错位，而不是材料不够多。所以产品必须先强在问题建模和假设树，而不是急于给结论。

### 3. 证据不等于判断

材料、证据、阶段性判断、最终结论必须是不同对象。否则最后只会得到信息堆积，而不是研究资产。

### 4. 分支必须被保留

高质量研究不是一路线性推进，而是不断比较、排除、修正。被放弃的分支也是资产，而不是噪音。

### 5. 结果应该是“过程资产”，不只是文档

一轮研究结束后，留下来的不应该只是一个结论页或报告，而应该是一套可以复用的研究结构。

## V1 研究资产定义与边界

### 什么是“研究项目”

研究项目是一次围绕单个高价值问题展开的工作单元。它有一个主问题、若干假设分支、不断演化的证据与判断，以及当前最佳结论。

### 什么是“研究资产包”

研究资产包是研究项目中被整理、发布并可复用的成果集合。它至少包含：

- 主研究问题
- 假设树与关键分支
- 已接受证据及来源指纹
- 被确认的判断
- 当前结论与下一步建议
- 分支复盘信息与版本元数据

### 工作区、资产包与发布边界

- **工作区（Workspace）**：用于进行草稿探索、快速调整、试错和分支扩展。`Workspace` 本身不会变成“已发布资产”，它只是发布动作的来源容器。
- **资产包（ResearchAssetBundle）**：从某个 `Workspace` 的一个发布快照生成的正式成果，面向复用、共享和审计。
- **发布边界**：发布动作会从 `Workspace` 生成一个新的 `ResearchAssetBundle` 版本；已发布资产包不允许直接覆写，只能生成新版本或被标记为 `Superseded`。

### V1 发布快照包含项

V1 的 `ResearchAssetBundle` 发布快照固定包含以下内容：

- 当前 `Workspace` 的单个主 `ResearchQuestion`
- 全部 `Hypothesis / SubQuestion`，包括 `Active`、`Parked`、`Superseded` 分支
- 被 `Confirmed Judgment` 引用的 `Accepted Evidence`
- `Conclusion / NextStep`
- 发布摘要、版本元数据、checkpoint 引用、审计引用

V1 的发布快照明确不包含：

- `Collected`、`Reviewed`、`Rejected` 状态 `Evidence` 的正文内容
- 完整审计日志正文
- 未被结论链路引用的临时草稿结论

说明：

- `Parked` / `Superseded` 分支保留在发布快照中，是为了支持复盘与组织记忆
- 审计和 checkpoint 在资产包里以引用形式存在，用于追溯，不作为正文内容随包复制

### 持久化粒度

- **节点级持久化**：问题、假设、证据、判断、结论节点的增删改查都自动保存。
- **分支级检查点**：每次重要分支决策、收敛或放弃时记录检查点，便于回放和比较。
- **资产包级版本快照**：每次发布形成不可变版本，用于共享、审计和跨项目复用。

### Checkpoint 最小模型

V1 单独定义 `Checkpoint` 对象，不把它等同于审计日志。

- **必填字段：** `id`、`workspaceId`、`triggerType`、`summary`、`snapshotRef`、`createdAt`、`actorId`
- **作用：** 作为工作区中关键研究节点的“可回放里程碑”
- **与审计的关系：** 审计日志记录细粒度动作；`Checkpoint` 记录少量高价值里程碑，可从 UI 中直接回看

V1 只支持 4 类触发点：

- `hypothesis_parked_or_superseded`
- `judgment_confirmed_or_superseded`
- `workspace_ready_for_publish`
- `asset_bundle_published`

### 共享范围与导出边界

- 个人工作区：指未共享的 `Workspace`，默认仅 `Owner` 可见和可编辑
- 团队工作区：`Workspace` 归属某个团队，`Owner` 可邀请同团队成员以 `Editor / Viewer / Auditor` 身份访问
- 团队资产：已发布 `ResearchAssetBundle` 团队内可见，可按角色查看、评论或复用；若要继续修改，必须克隆到新 `Workspace`
- 跨团队共享：默认只读，需要明确授权
- 导出：支持导出为报告摘要、结构化资产快照或只读链接

### 版本化与审计边界

- 草稿阶段允许覆盖编辑
- 已发布版本不允许原地覆写，只能基于新版本继续演进
- 以下动作必须留痕：发布、撤回发布、权限变更、证据核验、关键置信度变更、导出、跨团队共享

## 第一版信息架构与产品结构

### 全局信息架构

V1 至少包含以下页面或导航入口：

- **资产库 / 项目列表**：查看个人项目、团队资产、模板和最近复用记录
- **研究工作区**：主画布、右侧检视器、持续结果区三栏联动
- **发布页**：检查是否满足发布条件、填写摘要、生成资产包版本
- **权限与审计页**：查看角色、共享范围和关键操作日志

### 工作区三层结构

#### 第一层：主画布，负责呈现研究过程

主画布不是通用白板，而是只承载以下核心对象：

- 研究问题
- 子问题 / 假设
- 证据
- 判断
- 结论 / 下一步

#### 第二层：右侧检视器，负责理解单个对象

点击任意节点，右侧检视器展示：

- 这个对象是什么
- 它依赖什么
- 它支持或反驳哪个判断
- 当前状态、可信度、作者与时间戳
- 推荐下一步动作

#### 第三层：持续收敛的结果区

结果区不应该等最后才出现，而应该持续显示：

- 当前最佳结论
- 当前最强理由
- 当前未解决风险
- 当前下一步建议

## 对象字典（Object Dictionary）

### 1. ResearchQuestion

- **作用：** 研究项目的根问题
- **必填字段：** `id`、`title`、`problemStatement`、`owner`、`team`、`status`、`createdAt`、`updatedAt`
- **允许操作：** 创建、编辑、链接假设、归档
- **关键约束：** 一个研究项目只能有一个主 `ResearchQuestion`

### 2. Hypothesis / SubQuestion

- **作用：** 对主问题进行结构化拆解的分支
- **必填字段：** `id`、`title`、`type`、`status`、`confidence`、`parentId`、`owner`、`createdAt`、`updatedAt`
- **允许操作：** 创建、拆分、链接证据、搁置、恢复、归档
- **关键约束：** 必须挂在 `ResearchQuestion` 或另一个 `Hypothesis` 之下

### 3. Evidence

- **作用：** 支撑或反驳假设与判断的材料单元
- **必填字段：** `id`、`title`、`sourceType`、`sourceFingerprint`、`sourceLocator`、`excerpt`、`capturedAt`、`collector`、`status`
- **允许操作：** 添加、编辑、链接、复核、接受、拒绝、导出引用
- **关键约束：** 每条证据必须带来源指纹与定位信息；没有来源的材料不能进入已接受状态；进入 `Reviewed` 后必须记录 `reviewedAt`、`reviewedBy`，进入 `Verified` 后必须记录 `verifiedAt`、`verifiedBy`，进入 `Accepted` 后必须记录 `acceptedAt`、`acceptedBy`

### 4. Judgment

- **作用：** 基于证据形成的阶段性专业判断
- **必填字段：** `id`、`statement`、`confidence`、`status`、`author`、`evidenceRefs`、`createdAt`、`updatedAt`
- **允许操作：** 创建、编辑、确认、替换、链接结论
- **关键约束：** 每个 `Judgment` 必须引用至少一条 `Evidence`

### 5. Conclusion / NextStep

- **作用：** 当前收敛出的结论与建议动作
- **必填字段：** `id`、`summary`、`status`、`owner`、`judgmentRefs`、`openRisks`、`nextActions`、`updatedAt`
- **允许操作：** 起草、编辑、标记可发布、发布、归档、导出
- **关键约束：** 结论必须引用一个或多个已确认的 `Judgment`

### 6. Workspace

- **作用：** 承载一次研究项目的草稿过程与运行中分支
- **必填字段：** `id`、`title`、`team`、`status`、`rootQuestionId`、`owner`、`createdAt`、`updatedAt`
- **允许操作：** 创建、编辑、检查点、发起发布、归档
- **关键约束：** `Workspace` 归属单一团队，但默认只有 `Owner` 可编辑；经授权后，同团队 `Editor / Viewer / Auditor` 可按权限访问；`Workspace` 不能作为可复用资产直接共享给其他团队；当进入 `ReadyForPublish` 后，工作区内容冻结，只允许处理发布摘要或退回 `Active`

### 7. ResearchAssetBundle

- **作用：** 承载从 `Workspace` 生成的已发布成果与版本
- **必填字段：** `id`、`title`、`team`、`version`、`status`、`sourceWorkspaceId`、`rootQuestionId`、`permissions`、`publishSummary`、`createdAt`、`updatedAt`
- **允许操作：** 发布、克隆、模板化、跨项目导入、导出、归档
- **关键约束：** `version` 使用单调递增整数；若某版本被新版本替代，则记录 `supersededByVersion`；已发布资产包不可原地覆写；继续编辑必须创建新版本或克隆到新工作区

## 状态机与关系规则

### 最小状态集合

- **Workspace：** `Draft -> Active -> ReadyForPublish -> Archived`
- **ResearchAssetBundle：** `Published -> Superseded -> Archived`
- **Hypothesis / SubQuestion：** `Draft -> Active -> Parked | Superseded -> Archived`
- **Evidence：** `Collected -> Reviewed -> Verified -> Accepted | Rejected`
- **Judgment：** `Proposed -> Confirmed -> Superseded`
- **Conclusion / NextStep：** `Draft -> Ready -> Published -> Archived`

### 状态语义补充

- `Workspace.Draft`：项目刚创建，问题与分支仍在搭建
- `Workspace.Active`：项目进入正常研究推进
- `Workspace.ReadyForPublish`：发布前冻结检查点，只允许补充发布摘要、执行校验或退回修改
- `ResearchAssetBundle.Published`：已经发布、可复用、不可覆写
- `ResearchAssetBundle.Superseded`：已有更新版本替代，但仍可审计与引用
- `ResearchAssetBundle.Archived`：发布被撤回或下线后进入归档；仍保留审计与历史查看，但不再允许新复用或新引用
- `Hypothesis.Parked`：该分支当前不继续推进，可通过 `parkReason=disproven|deprioritized|merged` 记录原因，后续可以恢复
- `Hypothesis.Superseded`：该分支已被更优分支替代，不再恢复到 `Active`
- `Judgment.Superseded`：该判断曾经成立，但已被更新判断替代
- `Evidence.Reviewed`：已检查基础元数据和关联关系，但尚未完成人工核验
- `Evidence.Verified`：已由人工确认来源真实性与定位正确
- `Evidence.Accepted`：已被当前研究正式纳入论证链，可供 `Judgment` 引用

### 提升路径规则

系统必须强制以下提升路径：

- `Evidence -> Judgment -> Conclusion`
- 没有 `Verified` 证据，不能将证据标记为 `Accepted`
- 没有已接受证据，不能确认判断
- 没有已确认判断，不能发布结论

### 分支与引用规则

- 被搁置的 `Hypothesis` 分支使用 `Parked`；被更优分支替代的使用 `Superseded`
- `Hypothesis.Parked` 可以恢复到 `Active`，`Hypothesis.Superseded` 不可恢复，只能归档
- 跨资产引用默认只读，若要继续编辑，必须导入或克隆到当前工作区
- 结论只能引用当前版本内有效的判断；跨版本引用必须显式记录来源版本
- 撤回发布时，当前 `ResearchAssetBundle` 转为 `Archived`；已有历史引用继续保留并显示“已撤回”标记，但新项目不能再将其作为复用来源

## 最小枚举与格式附录

### sourceType

V1 只支持以下来源类型：

- `web`：网页或在线文章
- `pdf`：PDF 文档
- `doc`：文档文件或在线文档导出
- `sheet`：表格
- `meeting-note`：会议纪要
- `manual-entry`：人工录入内容

### sourceFingerprint

`sourceFingerprint` 是去重与追溯使用的稳定标识，V1 统一使用：

`sha256(sourceType + canonicalSourceId + sourceLocator + normalizedExcerpt)`

其中：

- `canonicalSourceId`：URL、文件 ID、文档 ID 或会议记录 ID
- `sourceLocator`：来源中的定位信息
- `normalizedExcerpt`：去空白后的引用片段

规则：

- 在 `Evidence` 进入 `Verified` 或 `Accepted` 后，`sourceType`、`sourceLocator`、`excerpt`、`sourceFingerprint` 视为冻结字段
- 若必须修改冻结字段，系统必须复制出新的 `Evidence` 对象，并将新对象状态重置为 `Collected`
- 原对象保留原始指纹和审计记录，不做原地覆写

### sourceLocator

V1 统一存储为字符串，格式最小化如下：

- `web`：`url#fragment`
- `pdf` / `doc`：`page:<n>` 或 `page:<n>-<m>`
- `sheet`：`sheet:<name>!<range>`
- `meeting-note` / `manual-entry`：`section:<id>`

### confidenceLevels

V1 统一使用三级置信度：

- `low`
- `medium`
- `high`

规则：

- `Judgment.confidence` 和 `Hypothesis.confidence` 只能由 `Owner` 或 `Editor` 修改
- Agent 可以提出置信度建议，但不能直接写入最终值
- 每次置信度修改都必须写入审计日志

- `Owner` 负责高风险动作和权限管理
- `Auditor` 只读访问审计信息，不参与内容编辑

### 协作动作权限表

| 角色 | 评论 | `@` 提问 | 请求复核 | 请求批准发布 | 查看请求队列 |
| --- | --- | --- | --- | --- | --- |
| Owner | 是 | 是 | 是 | 是 | 是 |
| Editor | 是 | 是 | 是 | 是 | 是 |
| Viewer | 是 | 是 | 否 | 否 | 否 |
| Auditor | 否 | 否 | 否 | 否 | 是 |

约束说明：

- `Viewer` 可以参与异步反馈，但不能发起流程性动作
- `Auditor` 可以查看请求与处理结果，以支持审计追踪，但不参与协作内容本身

### 协作请求最小模型

V1 只做一种协作对象：`ReviewRequest`

- **必填字段：** `id`、`type`、`status`、`targetType`、`targetId`、`requesterId`、`assigneeId`、`createdAt`、`updatedAt`
- **类型：** `evidence-review`、`publish-approval`
- **状态：** `Open -> Resolved | Rejected`

规则：

- `ReviewRequest` 必须绑定到具体 `Evidence` 或 `Workspace`
- `evidence-review` 只能绑定 `Evidence`
- `publish-approval` 只能绑定已进入 `ReadyForPublish` 的 `Workspace`
- V1 不做独立通知中心，请求只在工作区请求队列中展示

### 发布检查清单

一次发布至少要满足以下硬规则：

1. 存在且仅存在一个主 `ResearchQuestion`
2. 至少有 `1` 条 `Confirmed Judgment`
3. 每条被 `Conclusion` 引用的 `Judgment` 都必须引用至少 `1` 条 `Accepted Evidence`
4. 每条 `Accepted Evidence` 都必须包含非空的 `sourceFingerprint`、`sourceLocator`、`capturedAt`、`collector`
5. `Conclusion / NextStep` 必须处于 `Ready` 状态，且 `judgmentRefs`、`openRisks`、`nextActions` 非空
6. `Workspace` 必须先进入 `ReadyForPublish`，并在发布成功后生成新的 `ResearchAssetBundle.version`
7. 允许存在 `Parked` 分支；若仍有 `Active` 分支，则这些分支必须在 `openRisks` 中被明确提及
8. 不允许仍处于 `Reviewed` 但未完成 `Verified` 或 `Rejected` 的 `Evidence` 进入发布范围

### 审计日志最少字段集合

每条审计日志至少包含：

- `timestamp`
- `actorId`
- `actorRole`
- `action`
- `targetType`
- `targetId`
- `assetVersion`
- `before`
- `after`
- `reason`
- `details`

其中 `details` 至少要能记录本次动作的关键上下文，例如来源指纹、置信度变更、共享目标范围、发布摘要。

## 第一版核心工作流与主用户旅程

### 核心工作流

第一版围绕一条清晰的研究流程展开：

1. 从一个高价值问题开始
2. 把问题拆成假设树
3. 让分支并行推进
4. 把证据提升为判断
5. 收敛成结论，同时保留备选路径
6. 将整轮研究发布为团队资产包

### V1 主用户旅程

1. **新建研究项目**
   产物：`ResearchQuestion`、初始 `Workspace`
   落盘点：创建项目与根问题
2. **生成并调整假设树**
   产物：一组 `Hypothesis / SubQuestion`
   落盘点：树结构、分支说明与负责人
3. **并行推进分支与收集证据**
   产物：待验证问题、证据节点、分支检查点
   落盘点：证据来源、片段、抓取时间、关联假设
4. **复核证据并提升为判断**
   产物：`Judgment`
   落盘点：证据引用、判断置信度、复核记录
5. **收敛当前结论与下一步**
   产物：`Conclusion / NextStep`
   落盘点：支撑判断、未决风险、下一步行动
6. **发布为团队资产包**
   产物：版本化 `ResearchAssetBundle`
   落盘点：版本号、发布摘要、权限范围、审计记录
7. **在下一轮研究中复用**
   产物：新项目中的模板、克隆分支或导入对象
   落盘点：复用来源与复用关系

## Agent 边界与确认点

### Agent 可以做什么

- 建议问题拆解与假设树结构
- 提出证据检索任务和反证方向
- 总结多条证据之间的支持与冲突
- 起草阶段性判断与结论候选
- 提醒缺失证据、冲突证据与高风险未决项

### Agent 不能做什么

- 编造来源或伪造证据
- 替代人类做最终专业判断
- 自动发布资产包
- 自动删除已发布资产
- 自动变更权限或跨团队共享设置

### Agent 必须请求人工确认的动作

- 发布资产包
- 将 `Evidence` 标记为已核验或已接受
- 调整关键判断的置信度等级
- 导出、跨团队共享或外发研究资产

### 证据溯源与可追溯要求

每条 `Evidence` 至少要绑定：

- 来源类型
- 来源指纹或唯一标识
- 抓取或录入时间
- 片段引用或定位信息
- 采集人或采集 Agent

## 最小异步协作与复用模型

### 最小协作动作

第一版支持最小但必要的异步协作：

- 评论
- `@` 提问
- 请求复核
- 请求批准发布

### 共享方式

- 未共享的 `Workspace` 默认个人私有
- 获授权的团队工作区支持团队内只读 / 可评论 / 可编辑
- 跨团队只读分享

### 复用方式

- 从模板新建研究项目
- 克隆完整资产包
- 从历史资产包导入节点或分支
- 复制结论结构但重新挂接本轮证据

### 第一版明确不做

- 实时光标 Presence
- 多光标实时编辑
- 复杂冲突合并

## 第一版必须做强的能力

第一版必须真正做强下面这些能力：

- 研究问题建模
- 子问题 / 假设树拆解
- 多分支研究推进
- 证据与判断分离
- 持续收敛的结论区
- 结果沉淀为团队资产包
- Agent 辅助推进分支、补证据、整理判断和后续动作

## 第一版必须刻意不做的能力

第一版必须刻意克制，避免失焦。

- 不做通用白板
- 不做实时多人 Presence、多光标编辑和复杂协作冲突处理
- 不做全企业知识中台
- 不做重型跨系统执行编排
- 不做泛行业全覆盖
- 不做以模型 / 套餐 / Agent 市场为主的商业包装

这些以后都可能要做，但不应该进入第一阶段。

## 企业部署、数据边界、权限与审计假设

### 部署与数据边界

- V1 以私有部署 / 内网优先为主假设，公有云 SaaS 多租户运营能力不进入首轮实现范围
- 默认按单企业租户隔离
- 跨租户数据默认完全隔离
- 优先支持私有部署 / 内网优先场景
- 数据驻留、保留期和备份策略按企业策略配置

### 身份与组织边界

V1 的身份体系固定为：

- 应用内本地用户
- 应用内团队成员管理
- 应用内角色分配

V1 明确不做：

- 企业 SSO
- LDAP / AD 同步
- 外部身份源自动映射

这些能力进入 V2，再与企业真实身份体系对接。

### 角色与权限

第一版至少包含四类角色：

- `Owner`：管理项目、发布和权限
- `Editor`：编辑工作区内容，发起发布
- `Viewer`：查看与评论
- `Auditor`：查看资产与审计记录，但不修改内容

### 审计日志范围

以下动作必须进入审计日志：

- 资产变更
- 权限变更
- Agent 触发的动作
- 证据引入、复核与接受 / 拒绝
- 发布、导出、跨团队共享

### 合规与安全假设

- 默认不使用客户数据训练通用模型
- 对敏感来源访问采用企业控制策略
- 导出内容支持可选水印或指纹

## V1 成功指标与失败信号

### 成功指标

第一版至少观察以下指标：

- 从新建问题到形成第一版假设树的中位时间 `<= 30 分钟`
- 每个活跃团队每月发布的资产包数量 `>= 4`
- 已发布资产在 60 天内被复用的比例 `>= 30%`
- 已接受证据的可追溯覆盖率 `>= 95%`
- 团队级 8 周留存 `>= 60%`

### 失败信号

如果出现以下现象，说明 V1 方向有问题：

- 用户把它当成普通聊天工具，而不是研究工作台
- 工作区很多，但正式发布率持续低于 `20%`
- 结论很多，但证据链缺失严重，已接受证据可追溯覆盖率低于 `80%`
- 历史资产几乎不被复用，60 天复用率接近 `0`

## 产品壁垒

这个产品的壁垒建议分成四层来理解。

### 1. 工作流壁垒

它深度嵌入战略研究工作流，而不是泛用 AI 使用场景。

### 2. 资产复利壁垒

每一轮研究都会沉淀为可复用资产，而不是一次性答案。

### 3. 组织记忆壁垒

团队不再只保留结果，而是保留：

- 问题如何定义
- 哪些假设被提出
- 哪些证据支持或反驳
- 为什么最终选了这个方向

### 4. 界面心智壁垒

大画布和工作空间感可以帮助产品在感知层拉开差异，但它必须服务于更深的研究资产壁垒，而不能取代它。

## 投资人叙事

最强的投资人叙事建议是：

> 这不是一个更漂亮的 AI 助手，而是一套把战略研究过程转化为组织资产的系统。

更完整的增长路径可以这样讲：

1. 先用一个窄而深的研究资产工作台打透企业战略 / 战投团队
2. 再扩到其他高价值研究团队，如咨询、投研、产业研究、政策分析
3. 最后把资产图谱、Agent 推进层、权限、模板和记忆系统平台化，长成更广义的企业 Agent Workspace

## 专业专家叙事

最强的专家侧叙事建议是：

> 这个产品不是替你下判断，而是把判断如何形成沉淀下来。

专家愿意长期留下来的前提，是产品真正做到：

- 帮助把问题建对
- 让证据和判断严格区分
- 保留被放弃的路径
- 支持下一轮研究直接复用
- 保持专家对最终解释权和判断权的主导

## 推荐的一句话定位

建议使用下面这句作为正式定位：

> 企业战略 / 战投团队的研究资产工作台。

如果需要更完整的一句产品介绍，可以使用：

> 一款漂亮的 AI 原生研究工作空间，帮助企业战略 / 战投团队把研究过程沉淀成可复用的组织资产。
